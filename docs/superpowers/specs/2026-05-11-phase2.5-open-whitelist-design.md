# llamactl Phase 2.5: Open the Whitelist — Design Spec

**Status:** Approved 2026-05-11 (this session).
**Builds on:** Phase 2 (commits `873d464..0996a29`, merged via `c101b49` on 2026-05-11).
**Spec change:** PRD v1.5 §4 ("Curated model whitelist (v1)") and §3 (`add`, `search` commands). See §10 for the PRD-amendment text.

---

## 1. Goals

Eliminate the friction caused by Phase 2's closed whitelist: every new model on HuggingFace currently requires a code-change-and-rebuild to install. Phase 2.5 keeps the curated short-id table as a convenience layer but lets `add` accept any HuggingFace GGUF repo path directly.

Acceptance:

1. `llamactl add qwen2.5-7b-instruct` works exactly as today.
2. `llamactl add Qwen/Qwen3-8B-Instruct-GGUF --quant Q4_K_M` downloads, verifies SHA, reads the GGUF header for `ParamsB`/`Arch`, and writes per-tool metadata.
3. `llamactl search qwen3` returns the Qwen3 GGUFs (no longer filtered out).
4. `llamactl search qwen2.5` shows both preferred entries (marked `*`) and non-preferred GGUFs.

## 2. Non-goals

- Migrating existing Phase 2 metadata files. Legacy entries lack `params_b`/`arch`; `list` shows blank/`—` in those columns and we never touch the file.
- Reading `config.json` siblings on HF. GGUF header parsing is sufficient and authoritative.
- Multi-shard GGUFs. Still rejected, same logic as Phase 2.
- A `--params-b` / `--arch` flag set on `add`. The GGUF header is the source of truth post-download.
- Removing short-ids. They stay; the whitelist is renamed but not deleted.
- PRD `serve`/launchd (Phase 3 scope).

## 4. `internal/gguf` package

### 4.1 Format reference (GGUF v2/v3)

```
offset 0:   magic = "GGUF" (4 bytes)
offset 4:   version u32 LE
offset 8:   tensor_count u64
offset 16:  kv_count u64
offset 24:  kv_entries[kv_count]
            each: { key_len u64, key bytes, value_type u32, value bytes }
```

All integers little-endian (GGUF spec mandates LE).

### 4.2 API

```go
package gguf

import "errors"

var (
    ErrBadMagic            = errors.New("GGUF magic not present")
    ErrUnsupportedVersion  = errors.New("unsupported GGUF version")
    ErrUnsupportedGGUFType = errors.New("unsupported GGUF value type")
)

type Header struct {
    Version       uint32
    TensorCount   uint64
    Architecture  string // value of "general.architecture", e.g. "llama", "qwen2", "qwen3"
    ParamsCount   uint64 // value of "general.parameter_count"
    ContextLength uint64 // value of "<Architecture>.context_length"; 0 if absent
}

// ReadHeader opens path, reads the GGUF header, and returns the three
// metadata fields we care about. Other keys are skipped. Returns a typed
// error for bad magic, unsupported version, or unparseable value types.
func ReadHeader(path string) (Header, error)
```

### 4.3 Implementation notes

- Read first 1 MiB into memory. Most GGUF headers are well under 100 KiB; 1 MiB is comfortable headroom.
- Supported versions: 2 and 3. Anything else → `ErrUnsupportedVersion`.
- Walk metadata KVs sequentially. Collect the three target keys when seen; otherwise advance the read cursor past the value.
- Supported value types: u8/i8/u16/i16/u32/i32/u64/i64/f32/f64/bool/string. Arrays and nested types → `ErrUnsupportedGGUFType` with byte offset in the error message. We don't need them for the three keys.
- The architecture-specific key (e.g. `qwen2.context_length`) is only read after `general.architecture` is found; do a second pass through collected KVs if order isn't guaranteed (GGUF spec doesn't guarantee order).

## 7. `cli/add` flow

```
1. Parse input:
     contains "/"  → HF-path mode (input is "Org/Repo")
     else           → preferred-id mode (input is "qwen2.5-7b-instruct")

2. PREFERRED-ID MODE (unchanged from Phase 2):
     - models.LookupOrSuggest(input) → Model, or ErrUserError listing short-ids.
     - Auto-bootstrap hardware.json (existing).
     - Quant: --quant override OR QuantSelector.Select(model, hw, ctx).
     - hf.RepoInfo + findQuantFile → SHA, size, filename.
     - Dedupe check → Downloader.Get if needed.
     - Persist metadata { ID=model.ID, ParamsB=model.ParamsB, Arch=model.Arch, … }.

3. HF-PATH MODE (new):
     a. If --quant is empty:
          - hf.RepoInfo to enumerate available quants.
          - Return ErrUserError:
            "add <input> requires --quant; available in <Org/Repo>: Q4_K_M, Q5_K_M, …"
     b. Validate --quant ∈ models.PreferenceOrder.
     c. hf.RepoInfo + findQuantFile → SHA, size, filename.
     d. Derive ID:
          basename := path.Base(input)        // "Qwen3-8B-Instruct-GGUF"
          lower   := strings.ToLower(basename) // "qwen3-8b-instruct-gguf"
          id      := strings.TrimSuffix(lower, "-gguf")  // "qwen3-8b-instruct"
        Case-insensitive by virtue of lowercasing first, then trimming the
        lowercased suffix. Repos without "-GGUF" in their name yield the
        bare lowercased basename.
     e. destPath = <SharedModelsDir>/<derived-id>/<quant>.gguf.
     f. Dedupe check by on-disk SHA (existing fast path).
     g. Downloader.Get (unchanged).
     h. header, err := gguf.ReadHeader(destPath).
        If err != nil:
          - fmt.Fprintf(d.Stderr, "llamactl: warning: could not read GGUF header (%v); ParamsB/Arch omitted\n", err)
          - Proceed with header = Header{}.
     i. paramsB := int(header.ParamsCount / 1e9)  // floor; 7,615,616,512 → 7.
     j. arch := archFromGGUF(header.Architecture).
     k. Persist metadata { ID=<derived-id>, ParamsB=paramsB, Arch=arch, … }.

4. Both modes:
     fmt.Fprintf(d.Stdout, "installed %s (%s, %s) -> %s\n", id, quant, humanFileSize(size), destPath)
```

`archFromGGUF` lives in `cli/add.go`:

```go
func archFromGGUF(s string) models.Arch {
    switch s {
    case "llama":   return models.ArchLlama3
    case "qwen2":   return models.ArchQwen25
    case "mistral": return models.ArchMistral
    default:        return models.Arch(s)
    }
}
```

For unknown archs (`qwen3`, `gemma`, etc.) we store the string verbatim. The selector won't recognize them, but the HF-path flow requires explicit `--quant`, so the selector isn't called on these models in Phase 2.5.

### 7.1 Collision case

If a derived ID matches a preferred ID (e.g. user types `add Qwen/Qwen2.5-7B-Instruct-GGUF` instead of `add qwen2.5-7b-instruct`), the derived destPath equals the preferred destPath. Same file, same SHA, same metadata path. The metadata write overwrites cleanly — same data either way.

## 8. `cli/search` flow

```
1. hits := hf.Search(query)  (no filter)
2. Cap to first 25 hits.     (--limit deferred to Phase 3 if needed)
3. Build reverse index: byRepo := map[HFRepo]Model from PreferredIDs.
4. For each hit:
     if m, ok := byRepo[hit.ID]; ok → preferred row {*, m.ID, m.ParamsB, …}
     else                          → unknown row { " ", hit.ID, "?", …}
5. Sort: preferred entries first (alphabetical by short-id), then non-preferred
   (by HF Downloads descending — already the HF API's default).
6. For each row, fetch RepoInfo (7d cache absorbs the cost) to populate the
   QUANTS column. For non-preferred rows, skip RepoInfo only if cache miss
   AND a future --no-quants flag says so; not in this phase.
7. tabwriter output:

      MODEL                                   PARAMS  QUANTS              REPO
   *  qwen2.5-7b-instruct                     7B      Q5_K_M,Q4_K_M,Q2_K  Qwen/Qwen2.5-7B-Instruct-GGUF
   *  qwen2.5-coder-7b                        7B      Q5_K_M,Q4_K_M,Q2_K  Qwen/Qwen2.5-Coder-7B-Instruct-GGUF
      Qwen/Qwen3-8B-Instruct-GGUF             ?       Q5_K_M,Q4_K_M       Qwen/Qwen3-8B-Instruct-GGUF
```

MODEL column shows the short-id for preferred entries, the full HF path for non-preferred.

## 10. Error model

- `ErrUserError` (exit 2): unknown short-id, missing `--quant` for HF path, `--quant` not in `PreferenceOrder`, quant filename absent on HF, multi-shard GGUF.
- Generic error (exit 1, `llamactl:` prefix): HF unreachable, fs I/O, atomic-rename failure.
- `gguf.ReadHeader` failure: **not** fatal. Warn to stderr; persist metadata with zero `ParamsB` and empty `Arch`.

- `search` returns un-filtered HF results with preferred entries annotated.

## 13. Risks

- **GGUF format drift.** GGUF v4 exists in unreleased llama.cpp builds; if HF starts publishing v4 GGUFs in v1 timeframe, `ReadHeader` errors out cleanly (`ErrUnsupportedVersion`) and the warn-and-proceed path persists metadata without `ParamsB`/`Arch`. Acceptable degradation.
- **HF rate limiting on search.** Removing the whitelist filter means more `RepoInfo` calls per search (one per result row to enumerate quants). 7-day repo cache absorbs repeated queries; cold-cache 25-hit searches do 25 GETs. Mitigation: `HF_TOKEN` raises the anonymous limit; we documented it in Phase 2 README.
- **Derived ID collisions across users.** Two different HF repos that strip to the same lowercase basename (e.g. `Foo/Bar-GGUF` and `Baz/Bar-GGUF`) would clobber each other's metadata. Unlikely in practice; if it happens, user gets confused state and can resolve with `remove --purge`. Acceptable for v1.
- **Synthetic GGUF bytes in tests.** The test fixtures are tiny hand-built byte slices, not real model files. Risk: we encode the format wrong and tests pass on broken parsing. Mitigation: the integration test (§11.5) reads through `hf.Client` + `Downloader` + `gguf.ReadHeader` end-to-end, exercising the same code path that production uses on real files.
